# Contributing

This site is generated directly from the [AI4RA/prompt-library](https://github.com/AI4RA/prompt-library) repository. To add or change content, edit files in the repo and open a pull request.

## Adding a new component

1. Pick a slug (lowercase, hyphenated, descriptive — e.g., `nih-noa-extraction-udm`).
2. Copy `templates/new-component/` to `components/<slug>/`.
3. Edit the YAML frontmatter in `prompt.md`. Use only values from [the taxonomy](taxonomy.md) for `category`, `domain`, and `manifestations`. Propose additions to the taxonomy in the same PR.
4. Fill `README.md` (one-paragraph summary, inputs, outputs, manifestations, evals, provenance).
5. If the component has a structured output, add `schema.json` (JSON Schema draft 2020-12) and keep its `version` field in lockstep with the frontmatter version.
6. Add at least one eval case under `evals/cases/<case>/` with `metadata.yaml`, `input-source.md`, and an `expected.*` golden output.
7. Add a `CHANGELOG.md` entry for `1.0.0 — <date>`.

## Versioning

Semver, per component:

- **MAJOR** for output-contract breaks
- **MINOR** for backward-compatible additions
- **PATCH** for wording or clarity

Bump the version in *every* manifestation (`prompt.md`, `skill/SKILL.md`, `agent/AGENT.md`) and in `schema.json` when a component changes.

## Rebuilding this site locally

The site is generated by a Python script that reads `components/` and emits Markdown into `docs/`. Then MkDocs Material builds the final HTML.

```bash
pip install -r requirements-docs.txt
python scripts/build_docs.py
mkdocs serve
```

Open [http://127.0.0.1:8000/](http://127.0.0.1:8000/). `mkdocs serve` watches `docs/` for changes. If you edit a component, rerun `python scripts/build_docs.py` — MkDocs will pick up the regenerated pages on its next reload.

## CI

Every push to `main` rebuilds and redeploys the site via `.github/workflows/pages.yml`. Pull requests do not auto-deploy, but the same workflow can run on a branch for preview if the repo's Pages source is configured for it.

## Canonical UDM

For changes that have implications for the canonical Unified Data Model (new schema fields, new vocabulary, etc.), file a discussion issue on [`ui-insight/AI4RA-UDM`](https://github.com/ui-insight/AI4RA-UDM) and reference it in the library's CHANGELOG and PR.
